# Docs

Страницы документации в этом проекте пишутся в Markdown. Каждый файл `.md` VitePress превращает в отдельную страницу сайта.

- Официальная документация VitePress по Markdown: [vitepress.dev/guide/markdown](https://vitepress.dev/guide/markdown)
- Базовый синтаксис Markdown: [markdownguide.org/basic-syntax](https://www.markdownguide.org/basic-syntax/)

Ниже собраны основные варианты Markdown и VitePress Markdown Extensions. Для каждого пункта сначала показан синтаксис, затем результат. При необходимости одну страницу можно собирать из нескольких Markdown-файлов через include-механику VitePress.

## **Редактирование страниц через PageCMS**

Тексты документации лежат «в одном месте» вместе с проектом сайта. Менять их удобнее не в программе для кода, а в веб-редакторе **[PageCMS](https://pagescms.org/)** — откроется по ссылке **[app.pagescms.org](https://app.pagescms.org/)**. Вход — через тот же аккаунт **GitHub**, что и у приглашения в проект (логин почты/пароля PageCMS отдельно не нужен).

### 1. Как войти в админку {#pagecms-access}

Подойдёт любой из вариантов.

- **Через наш сайт с документацией.** Найди страницу входа для редакторов. Если сайт открыт у тебя на компьютере в режиме разработки — адрес обычно похож на `http://localhost:5173/site.ams.docs/admin/` (цифры порта может подсказать команда). На опубликованном сайте это путь **`/site.ams.docs/admin/`** после адреса хостинга. Страница **сама перебросит** на [app.pagescms.org](https://app.pagescms.org/) — там в списке проектов выбери **site.ams.docs** и при необходимости нажми **войти через GitHub**.

- **По приглашению по почте.** Когда тебя добавили в проект как **участника с правами правки**, приходит письмо с **ссылкой «принять приглашение»** — его нужно пройти. После этого снова зайди через [app.pagescms.org](https://app.pagescms.org/) или через **`/site.ams.docs/admin/`** на нашем сайте. Если в письме есть **прямая ссылка** в редактор — можно открыть и её.

### 2. Выбрать, **куда** сохранять черновик (ветку) {#pagecms-editing-branch}

Когда проект уже открыт, **до любых правок** посмотри **слева**: там выбирается **версия черновика**, в которую запишется твоё сохранение. Переключатель выглядит как список или выпадающий список **рядом с названием проекта**.

- Выбери **рабочую черновую версию**. У нас её чаще всего называют **`cms-edits`** (если команда не сказала другое имя).

- Не работай на **`main`** (иногда её называют «мастер», «линия выпуска»). Это **та версия, из которой собирают опубликованный сайт**. Всё, что ты сохранишь, попадёт именно в **ту строку черновика, которая сейчас выбрана**; если ошибочно оставишь **`main`**, ты можешь нарушить договорённости команды, а сохранение **может не пройти**.

Схема простая: **сначала выбрал черновик** → потом открыл язык (см. ниже) → потом создаёшь или правишь страницы и жмёшь **сохранить** → **в конце обязательно отправь правки на проверку** (п. 6 ниже: **создать пулл-реквест**).

**Как потом это попадёт на сайт для читателей.** В **`main` мы не правим руками «в лоб».** После сохранений в **`cms-edits`** (или в другую согласованную черновую ветку) изменения нужно **оформить запросом на слияние** (англ.: *pull request*). В редакторе для этого есть кнопка **Create PR to main** («создать запрос в основную линию») — подробнее в **п. 6**. Когда твои правки приняты и включены в **`main`**, обновится и опубликованный сайт. Подробности для техспециалистов — в файле **`CONTRIBUTING.md`** в корне проекта.

### 3. Выбрать **язык** материала {#pagecms-locale}

Слева, в блоке **Content**, перечислены **разделы по языку**. Выбери **тот язык**, на котором пишешь или правишь **сейчас**. В каждом таком пункте — **свой набор страниц**; сохранённые материалы попадут **только туда**.

Список может **расти**: помимо русского и английского команда может добавить ещё языки — они появятся в этом же блоке. Если видишь **непонятные названия**, попроси у команды подсказку.

На сейчас в нашем проекте обычно есть два пункта:

| Что видишь слева | Для чего это |
| ---------------- | ------------- |
| **Документация RU** | Страницы на **русском** языке |
| **Documentation EN** | Страницы на **английском** языке |

Подписи к полям в форме могут быть на разном языке: у русских страниц часто «**Заголовок**» и «**Содержимое**», у английских — **Title** и **Content**. Это нормально: так настроено в проекте.

### 4. Сделать **одну страницу** или **раздел со страницами внутри** {#pagecms-section-or-page}

- **Обычная отдельная страница** — без своей подпапки. В интерфейсе есть кнопка вроде **Add an entry** («добавить запись»): получишь **одну новую страницу** рядом с остальными на текущем уровне — подходит для статьи, справки, одного текста без подразделов.

- **Раздел-папка** — когда нужны **несколько вложенных страниц**. Обычно есть отдельная кнопка **создать папку** или **раздел** (иконка папки с плюсом). Внутри потом добавляют **дочерние** страницы (часто пункт **Add children entry** или «плюс» у строки раздела). Так делают, например, главу с несколькими подстатьями.

В нашем проекте у каждого **раздела-папки** по правилам должна быть **главная страница раздела** — файл **`index.md`**. Редактор обычно подсказывает это при создании раздела; **одной обычной странице с кнопки «добавить запись»** отдельный `index.md` не нужен.

Конкретные подписи кнопок могут чуть меняться в новых версиях редактора — ориентируйся на слова **добавить запись**, **папка**, **дочерняя страница**.

### 5. Имя файла, заголовок и содержимое — за что отвечает каждое поле {#pagecms-fields}

Когда открыта **форма страницы** (новой или уже существующей), ты обычно видишь **имя файла** (может быть не у всех экранов), **заголовок** и большое поле **содержимое**. Если подпись на английском (**Title**, **Content**, **Filename**) — смысл тот же, что описан ниже.

| Поле | Простыми словами, зачем оно |
| ---- | ---------------------------- |
| **Имя файла** (**Filename**) | Так страница **называется внутри проекта** как файл и как **кусочек будущего адреса** в браузере (часть ссылки без пробелов и «кракозябр»). У нас имя почти всегда **строится из заголовка** и заканчивается на `.md`; при создании иногда можно **подправить вручную**. Удобно **латиницей и через дефис**, например `instrukciya-po-ustanovke` — так ссылка читается предсказуемо. Переименовать файл позже **может сломать старые закладки** — лучше согласовать с командой. |
| **Заголовок** (**Title** / **Заголовок**) | **Название страницы для людей**: как она выглядит в списках в админке, во вкладке браузера, на сайте в навигации (если так настроено). Пиши обычным человеческим заголовком на нужном языке. Смена заголовка **не всегда** меняет имя файла — новый заголовок не обязан совпадать с старой ссылкой. |
| **Содержимое** (**Content** / **Содержимое**) | **Весь основной текст**, который читает посетитель: абзацы, списки, картинки, подзаголовки. Обычно есть **удобный режим набора** и переключатель **«исходник»** (там разметка Markdown) — что выбрать, решаешь ты; можно начать просто набирать текст. Вставлять ли сложные блоки («как в коде») — если неуверенно, лучше **спроси команду**, чтобы страница не перестала открываться. |

Краткая схема: **ссылочная техническая часть** — это в первую очередь **имя файла**, **видимое красивое название** — **заголовок**, **сам текст статьи** — **содержимое**.

### 6. После правок — нажми «создать пулл-реквест» {#pagecms-pull-request}

Когда ты **сохранил** страницу или несколько сохранений в своей черновой ветке (**`cms-edits`** и т.п.), этого **ещё недостаточно**, чтобы изменения попали на **живой сайт** для всех посетителей.

**Что нужно сделать:** в PageCMS найди блок действий (**Actions** или рядом с ним по левому краю) и нажми **Create PR to main** — по-своему это значит «**создать пулл-реквест**»: отправить твои правки из черновика **на общую проверку** и вливание в **основную** версию (**`main`**). Если такая заявка уже была, кнопка часто **обновит** её, а не создаст вторую с нуля.

**Зачем:** без этого шага правки так и остаются **только в черновой ветке** — на главном сайте их **не будет**, пока кто-то не нажмёт эту кнопку (или не сделает то же на GitHub вручную). После нажатия может открыться GitHub — это нормально; дальше обычно кто-то из команды **просматривает и подтверждает** слияние.

## Кастомные компоненты Vue в Markdown

В VitePress можно использовать свои Vue-компоненты прямо внутри `.md`-страницы. Для этого компонент нужно импортировать в блоке `<script setup>`, а затем вставить его как обычный тег.

В этом проекте есть компонент `DownloadFileButton`. Он принимает такие пропсы:

- `label` - текст на кнопке
- `href` - ссылка на файл

**Markdown**

````md
<script setup>
import DownloadFileButton from '../.vitepress/theme/components/ui/DownloadFileButton.vue'
</script>

<DownloadFileButton
  label="User manual.pdf"
  href="/files/user-manual.pdf"
/>
````

**Результат**

<DownloadFileButton
  label="User manual.pdf"
  href="/files/user-manual.pdf"
/>

## Кастомный компонент без пропсов

## Видео

Для видео в Markdown-странице используй HTML-тег `<video>`.  
Обычно файл кладут в `docs/public/videos`, а в `src` указывают путь от корня сайта.

**Markdown**

```html
<video controls width="720" preload="metadata">
  <source src="https://interactive-examples.mdn.mozilla.net/media/cc0-videos/flower.mp4" type="video/mp4" />
  Ваш браузер не поддерживает встроенное видео.
</video>
```

**Результат**
